记一次vue+Typescript项目重构

Posted on 2019-04-21 | Edited on 2020-04-04

项目现状

  1. 整个系统功能不完善。
  2. 代码耦合程度高，该抽离的组件不抽离，复用性差。
  3. 很多重复性代码和无效的逻辑判断代码，致使系统性能慢。
  4. 无用文件、无效代码、无效引入的包、定义的无效变量等不及时删除，工程文件越来越多越来越大，阅读性差，调试困难。
  5. 全局样式、全局变量满天飞，致使各组件样式相互影响、占用系统内存，降低系统性能。
  6. 代码规范性差，很难保证不同人的代码风格一致性，致使阅读性差，团队协作性差。
  7. 核心页面代码少则三四千多则五六千行，阅读性和维护性极差。
  8. 前后端定义的数据结构混乱，很多无用的数据传来传去，降低http请求和响应的时间。
  9. 路由结构定义有问题，从而导致无法使用子路由。
  10. 工程文件结构混乱
  11. …

存在的具体问题举例

1. 图设计模块

  1. 图设计中拖拽画布问题
    • 无法拖拽整个画布
    • 拖拽节点偶尔导致鼠标指针粘住问题
    • 点边表建图会导致极难绘制问题
  2. 偶尔出现连接线断开问题
  3. 节点连线用户体验差
  4. 全屏展示问题
  5. 图设计实验室模块卡顿明显
  6. 连线很难选中
  7. 未实现多边绘制、节点复制、label展示、有向边等功能
  8. 页面刷新问题

2.图分析模块

  1. 拖拽节点导致节点边和箭头绘制时间不一致
  2. 渲染效果不流畅
  3. 节点拖拽无效的力导向布局（影响性能）

解决方案

技术架构

1. vm框架-vue

考虑到公司前端大部分使用vue，并且也维护了一套适用于公司自己主题的vue组件库，因此选用vue框架，它是一个轻量级渐进式的MVVM库，简单易学、打出来的包比较小、国内用的人比较多，问题解决方案容易搜索，社区活跃、运行速度快。

2. 语言-Typescript

  • 为什么选用Typescript?
    • 提高后期代码的可读性、可以在编译阶段就发现大部分错误，这总比在运行时候出错好。
    • 类型系统实际上是最好的文档，大部分的函数看看类型的定义就可以知道如何使用了
    • 增强了编辑器和 IDE 的功能，包括代码补全、接口提示、跳转到定义、重构等。
    • TypeScript 是 JavaScript 的超集，.js 文件可以直接重命名为 .ts 即可
    • 即使不显式的定义类型，也能够自动做出类型推论
    • 可以定义从简单到复杂的几乎一切类型
    • 即使 TypeScript 编译报错，也可以生成 JavaScript 文件
    • 兼容第三方库，即使第三方库不是用 TypeScript 写的，也可以编写单独的类型文件供 TypeScript 读取
    • 并且在未来的vue3中也全面拥抱typescript
    • TypeScript 拥抱了 ES6 规范
    • 大部分第三方库都有提供给 TypeScript 的类型定义文件
  • 任何事物都有其两面性，上面那么多优点，那么它的缺点有哪些呢？
    • 有一定的学习成本，需要理解接口（Interfaces）、泛型（Generics）、类（Classes）、枚举类型（Enums）等前端工程师可能不是很熟悉的概念
    • 短期可能会增加一些开发成本，毕竟要多写一些类型的定义，不过对于一个需要长期维护的项目，TypeScript 能够减少其维护成本
    • 集成到构建流程需要一些工作量

总结： 因此从长远考虑项目的稳定性和可维护性，本项目建议使用Typescript进行开发。

3. 代码规范化插件-TSLint+Prettier

1. Prettier

Prettier 是一个支持多种文件格式（包括javascript、css、less、sass、typescript、markdown等）的样式格式化工具。它会删除代码里的原始样式，并确保输出的代码风格一致。优点如下：

  • 没有繁复的风格可配置项，它拥有一个近乎最佳实践总结出来的代码风格输出，符合绝大多数人的习惯（有点类似 Standard 的理念，但是又不那么极端，还是有少量的配置项，比如缩进空格数、tab或者空格缩进等）。
  • 支持制定行长，并针对所设置的行长对代码做样式优化
  • 支持众多编辑器，可轻松配置利用git hooks，支持git代码提交自动格式化
2.TSLint

TSLint是可扩展的静态分析工具，用于检查TypeScript代码的可读性，可维护性和功能性错误。它在现代编辑器和构建系统中得到广泛支持，并且可以使用自己的lint规则，配置和格式化程序进行自定义。

3. 项目组织结构
  • 使用文件夹管理组件, 样式文件和.vue文件分离，通过src属性引入，做到结构和表现型分离。
  • components文件夹中放入各个父组件的子组件，这样通过文件夹的组织结构就能清楚的知道组件间的子父级关系。
  • common文件夹中放各个模块或者组件中的公共组件、公共过滤器、公共混入方法和工具包等。如果文件比较小也可不必使用文件夹，按照场景决定有无。
  • 模块化路由配置，如果很多的页面或模块，常出现路由嵌套的情况，导致router.ts文件庞大，此时，建议以路由层级进行模块拆分。
  • 模块化组织Vuex状态，如果store很复杂时候，建议使用文件夹管理，并将store分割成模块。

代码规范化实践方案

案例: 当一个项目由多人一起维护的时候，假如A写了一个页面，B去维护A的页面加了一些逻辑，C去维护该页面再添加一些逻辑，当A再去维护这个页面的时候，可能就会发现已经不是当初他认识的代码了，函数也不是原来的语义，大部分的变量不知道干什么用的。那么A需要捋清楚B、C的逻辑，在B、C的基础上小心翼翼的编写新的需求，生怕哪一步写错，导致线上的代码出错。长此以往下去代码越来越难以维护，越来越少的人能看懂页面内的逻辑，糊墙式代码将页面堵得水泄不通！
因此代码规范化实践是非常重要的！！！

2. 模块化组织Vuex状态

Vuex 是一个专为 Vue.js 应用程序开发的状态管理模式。它采用集中式存储管理应用的所有组件的状态，并以相应的规则保证状态以一种可预测的方式发生变化。

VUEX

  1. 在什么情况下使用vuex呢？
    • 需要跨组件共享数据、跨页面共享数据的时候（但是对于简单的子父级组件的传值推荐用 props 和 events 实现父子组件通信）
    • 需要跨多级组件传递数据
    • 需要持久化的数据（如登录后用户的信息等）
    • 当一个组件需要多次派发一个事件，并且有多个组件进行接收和调用的时候
  2. 如何管理store?
    如果store很复杂时候，建议使用文件夹管理，并将store分割成模块,放入各自的模块中，遵循各个模块或者各个组件的业务数据放入到各自的store模块中，如果是总的业务数据放到总的store中。
3. 始终为组件样式设置作用域

全局样式容易污染其他组件样式。在vue组件中一旦使用了全局的style，那么你必将陷入无限的梦魇，因为你根本不知道什么时候组件的样式就被全局样式污染了。因此，建议始终为组件样式设置作用域。如果需要在组件中想修改其它子组件内部的样式，请使用深度作用选择器。

4. 始终为列表渲染提供唯一的key值

key 的特殊属性主要用在 Vue 的虚拟 DOM 算法，在新旧 nodes 对比时辨识 VNodes。如果不使用 key，Vue 会使用一种最大限度减少动态元素并且尽可能的尝试修复/再利用相同类型元素的算法。使用 key，它会基于 key 的变化重新排列元素顺序，并且会移除 key 不存在的元素。

5. 拆分大文件

当你发现你的文件已经超过四五百行代码，那么你就应该考虑到是否需要拆分组件、是否需要抽离公共方法等。

6. 精简.vue文件methods

.vue文件中methods对象中除页面初始化方法外的所有非事件处理方法放入到mixin的methods对象中，如果mixin的文件过大，可以考虑按功能拆分成多个mixin文件或者是抽离组件。

10. 避免过度组件

可能有一种“就不应该出现原生HTML，只要出现了都要封装成组件”的疯狂想法，为了封装而封装，导致页面最后变得稀烂，光找组件就得找半天。

11. 时刻注意抽离冗余代码

如果发现相同的功能逻辑或者是方法多次使用,此时应该想到是否能够把该逻辑抽离到全局的通用流程中，比如：

  1. Vue-router的hooks
  2. Vue的mixin
  3. Axios的interceptors
  4. Vuex和localStorage的无缝结合
12. 命名规范
  • 语义化命名（变量名尽可能可以取合适的长度,能够容易理解意义就可以;做到看代码的人能够通过你的命名知道在干什么）
  • CSS命名建议使用中杠来命名：graph-analysis-container
  • TS建议驼峰命名法：getNodeList
13. 代码注释的重要性

几行代码也许很容易知道功能，但几百行的代码，多个文件，也许刚写的时候是知道功能与特性的，但是随着时间的流逝，对自己的代码越来越难以理解，更别说其他人了。代码的可读性巨差，极大影响编程开发效率。

14. 其它规范
  1. 能够使用箭头函数的地方尽量时候箭头函数，一方面简洁了语法，一方面避免了偶尔不得不增加变量去保存this
  2. 尽量避免使用any类型
  3. 不用的代码全部删除
  4. 一个文件最好只写一个组件，避免紧耦合，避免维护困难、git冲突
  5. 路由名称和组件名称保持一致
  6. 需要写注释的尽量写注释，提高可维护性
  7. 能写全等就写全等
  8. 样式不要写在template中，统一写在样式文件中
  9. … 此处省略一千字

关注性能

好的用户体验对于任何一个Web App都是一个非常重要的方面，想要拥有一个好的用户体验，网页的性能优化是必不可少的环节；特别是针对项目比较大、并发性比较高或者页面结构和业务逻辑比较复杂的应用，性能优化在其中体现出至关重要的一部分。下面是我个人针对本项目的一些优化建议：

  1. 时刻考虑是否需要用到节流和防抖函数
  2. 重新定义变量保存引用值，以降低引用查询次数
  3. for循环保存length值
  4. 减少重绘和重排
  5. 正确使用watch和compute
  6. 能使用异步组件的地方正确使用异步组件
    在大型应用中，我们可能需要将应用分割成小一些的代码块，并且只在需要的时候才从服务器加载一个模块。为了简化，Vue 允许你以一个工厂函数的方式定义你的组件，这个工厂函数会异步解析你的组件定义。Vue 只有在这个组件需要被渲染的时候才会触发该工厂函数，且会把结果缓存起来供未来重渲染
  7. 开启路由懒加载
    当打包构建应用时，JavaScript 包会变得非常大，影响页面加载。如果我们能把不同路由对应的组件分割成不同的代码块，然后当路由被访问的时候才加载对应组件，这样就更加高效了。
  8. …

总结

本实践方案添加了很多约束项，短时间内看提高了项目的复杂度，但是从长远来看：

  1. 提高团队协作能力
  2. 提高了后期的可维护性
  3. 减少开发中人为的bug
  4. 提高了系统的稳定性
  5. 提高了系统的整体性能

俗话说磨刀不误砍柴工，我相信只要严格根据以上的实践方案，以后关联图谱项目的性能、可维护性、鲁棒性都会有很大的提升。
